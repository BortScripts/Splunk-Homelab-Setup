# Log Sources

---

## Windows Application Log

```bash
[WinEventLog://Application]
disabled = 0
index = main
```

The Windows Application log records events generated by applications and services running on the Windows system. This is different from the Security log. The Security log focuses on authentication, logons, permissions, and audit events. The Application log focuses more on what installed programs, services, databases, agents, or applications are doing.

Examples of things that may appear here:
- Application crashes
- Service errors
- Software installation issues
- Agent failures
- Application-specific warnings
- Database or web server errors
- Backup software events
- Security tool errors
- Unexpected application behavior

## Why this log matters

The application log is useful because attackers do not only interact with Windows itself. They often interact with the software running on the machine. For example, if malware causes an application to crash, if a security agent stops working, or if a vulnerable application starts throwing errors, the Application log may show evidence of that activity. This log is also useful for trouble shooting your homelab because it can show whether services are failing, agents are crashing, or applications are behaving unexpectedly.

## SOC value

From a SOC perspective, the Application log is not always the first place you look for obvious attacker activity, but it can provide important supporting evidence during an investigation. 

For example, 
- Security log shows suspicious login
- Sysmon shows suspicious process
- Application log shows the endpoint agent crashed shortly after

That combination could indicate something more serious than a normal application error.

## Common Event IDs
- 1000 (Application error (crash))
- 1001 (Windows error reporting (crash reporting details))
- 1002 (Application hang)
- 1026 (.NET Runtime error)

## Windows Security Log

```bash
[WinEventLog://Security]
disabled = 0
index = main
```

The Windows Security log records security-related events generated by the Windows operating system. This includes authentication activity, account usage, privilege changes, and audit events. It is one of the most important log sources in any environment because it directly reflects user behavior and access to the system. 

Examples of things that may appear here:
- Successful logins
- Failed login attempts
- Account lockouts
- Privilege escalation events
- User account creation and deletion
- Group membership changes
- Logoff events
- Special privileges assigned to accounts

## Why this log matters

The Security log is critical because almost every attacker interaction with a system involves authentication in some way. Whether it is logging in, attempting to guess passwords, or escalating privileges, those actions are recorded here.

This log allows you to detect:
- Brute force attacks
- Unauthorized access
- Suspicious login behavior
- Privilege escalation
- Lateral movement attempts

Without this log, you would have very limited visibility into who is accessing your system and how.

## SOC value

From a SOC perspective, the Security log is one of the highest value log sources. It is often the starting point for investigations because it shows who accessed a system, when they accessed it, and whether that access was successful or failed.

For example,
- Multiple failed logins from a single IP (brute force attempt)
- Followed by a successful login (account compromise)
- Followed by admin privileges being assigned

That sequence strongly indicates malicious activity.

## Key Event IDs
- 4624 (Successful logon)
- 4625 (Failed logon)
- 4672 (Special privileges assigned (admin-level access))
- 4688 (Process creation (if enabled via audit policy))
- 4720 (User account created)
- 4726 (User account deleted)
- 4732/4733 (User added/removed from group)

## Windows System Log

```bash
[WinEventLog://System]
disabled = 0
index = main
```

The Windows System log records events generated by the operating system itself and core system components. This includes hardware events, driver activity, service status changes, and system-level operations. This log focuses on how the operating system is functioning, rather than user activity (Security log) or application behavior (Application log).

Examples of things that may appear here:
- System startup and shutdown events
- Service start/stop/restart events
- Driver loading and failures
- Hardware issues (disk, CPU, memory)
- Network interface changes
- Time synchronization events
- System errors and warnings

## Why this log matters

The System log is important because attackers often interact with or impact core system functionality. While it may not always directly show malicious intent, it provides critical visibility into how the system is behaving underneath everything else.

This log allows you to detect:
- Services being stopped or disabled (potential defense evasion)
- Unexpected system reboots (possible crash or forced restart)
- Driver-related anomalies
- Network or system instability
- Changes to critical system components

It is also extremely useful for troubleshooting your homelab when something is not working correctly.

## SOC value

From a SOC perspective, the System log is often used as supporting evidence during investigations, but it can also highlight important system-level anomalies.

For example,
- Security log shows suspicious login
- Sysmon show suspicious process execution
- System log shows a critical service stopped shortly after

That combination could indicate an attacker attempting to disable security controls or destabilize the system.

## Common Event IDs
- 6005 (Event log service started (system startup))
- 6006 (Event log service stopped (system shutdown))
- 6008 (Unexpected shutdown)
- 7000/7001 (Service failed to start)
- 7036 (Service state changed (started/stopped))
- 7045 (A new service was installed)

## Sysmon Operation Log

```bash
[WinEventLog://Microsoft-Windows-Sysmon/Operational]
disabled = 0
index = main
```

The Sysmon (System Monitor) Operational log is generated by the Sysmon tool from Microsoft. It provides detailed visibility into system activity that is not available in standard Windows logs. Unlike default Windows logs, Sysmon is specifically designed for security monitoring and logs detailed information about processes, network connections, file activity, and more.

Examples of things that may appear here:
- Process creation (including command line arguments)
- Network connections initiated by processes
- File creation and modification
- DLL/image loading
- Registry changes
- Process injection behavior (depending on config)
- Parent-child process relationships

## Why this log matters

Sysmon is one of the most powerful log sources for detection engineering because it shows what actually happened on the system, not just that something happened.

For example:
- Security log: shows a login occurred
- Sysmon: shows what that user actually executed

This log allows you to detect:
- Malicious process execution
- Suspicious command-line activity
- Living-off-the-land binaries (LOLBins)
- Network connections tied to specific processes
- Lateral movement techniques
- File drops and payload execution

Without Sysmon, your visibility into attacker behavior is significantly limited.

## SOC value

From a SOC perspective, Sysmon is often considered a high-value telemetry source because it enables detection of real attacker techniques rather than just surface-level activity.

For example,
- Security log shows successful login
- Sysmon shows powershell.exe -enc... executed
- Firewall log shows outbound connection

That combination clearly indicates malicious activity.

## Key Event IDs

Sysmon uses its own event ID structure. The most important ones include:
- 1 (Process creation)
- 3 (Network connection)
- 7 (Image loaded (DLLs))
- 8 (CreateRemoteThread (possible injection))
- 11 (File creation)
- 13 (Registry value set)
- 22 (DNS query)

## Windows Firewall Log

```bash
[monitor://C:\Windows\System32\LogFiles\Firewall\pfirewall.log]
disabled = 0
index = main
sourcetype = firewall
```

The Windows Firewall log (pfirewall.log) records network traffic that is allowed or blocked by the Windows Defender Firewall. Unlike event logs, this is a flat text log file that captures connection-level details.

Examples of things that may appear here:
- Allowed outbound connections
- Blocked inbound connections
- Source and Destination IP addresses
- Protocol used (TCP/UDP)
- Action taken (ALLOW or DROP)

## Why this log matters

The firewall log is critical because it provides visibility into network activity, which is a key part of attacker behavior. Attackers almost always interact with the network:
- Scanning ports
- Connecting to command-and-control (C2) servers
- Moving laterally
- Downloading payloads

While Sysmon can show network connections tied to processes, the firewall log provides a network-centric view of traffic, which helps validate and correlate activity.

## SOC value

From a SOC perspective, the firewall log is extremely useful for identifying network-based indicators of compromise.

For example,
- Firewall log shows one IP scanning multiple ports
- Sysmon shows a process initiating connections
- Security log shows login activity

That combination strongly suggests malicious behavior.

## PowerShell Operational Log

```bash
[WinEventLog://Microsoft-Windows-PowerShell/Operational]
disabled = 0
index = main
```

The PowerShell Operational log records detailed activity from PowerShell execution on the system. When Script Block Logging is enabled, this log captures the actual PowerShell commands and scripts that were executed, not just that PowerShell was opened. This is one of the most important logs for detecting modern attacks because PowerShell is heavily abused by attackers for execution, downloading payloads, and lateral movement.

Examples of things that may appear here:
- Full PowerShell command execution (including scripts)
- Encoded or obfuscated commands
- File downloads via PowerShell
- Remote command execution
- Use of built-in PowerShell functions for malicious activity
- Script execution from memory (fileless attacks)

## Why this log matters

PowerShell is one of the most commonly abused tools in Windows environments because:
- It is built into the operating system
- It is extremely powerful
- It can execute code directly in memory
- It is often trust and not blocked

Without Script Block Logging, you might only see that PowerShell was opened. With this log, you can see exactly what was executed, which is critical for detecting real attacker behavior. This log allow you to detect:
- Encoded commands (-enc, Base64 usage)
- File downloads
- Execution of remote scripts
- Obfuscated or hidden commands
- Use of malicious PowerShell techniques

## SOC value

From a SOC perspective, this is a high-value detection log because it exposes attacker techniques directly. 

For example, 
- Security log shows login
- Sysmon shows powershell.exe execution
- PowerShell Operational log shows the exact malicious command

That combination gives you full visibility into what the attacker actually did.

## Key Event IDs
- 4104 (Script Block Logging)
- 4103 (Module logging (commands executed with modules))

4104 is the most important because it contains the actual script or command content. 4103 can provide additional detail on how commands were executed.

## Task Scheduler Operational Log

```bash
[WinEventLog://Microsoft-Windows-TaskScheduler/Operational]
disabled = 0
index = main
```

The Task Scheduler Operational log records activity related to scheduled tasks on the system. This includes the creation, modification, execution, and deletion of scheduled tasks. Scheduled tasks are a built-in Windows feature that allows programs or scripts to run automatically at specific times or under certain conditions. 

Examples of things that may appear here:
- Creation of new scheduled tasks
- Modification of existing tasks
- Execution of scheduled tasks
- Deletion of tasks
- Task failures or errors
- Task triggers (time-based or event-based execution)

## Why this log matters

Scheduled tasks are a very common persistence mechanism used by attackers. Instead of running malicious code manually every time, an attacker can create a scheduled task that automatically executes their payload. This allows attackers to:
- Maintain persistence after initial access
- Execute malware at regular intervals
- Run code when a user logs in
- Trigger actions based on system events

Because scheduled tasks are legitimate Windows functionality, they can blend in with normal system activity.

## SOC value

From a SOC perspective, this log is highly valuable for detecting persistence and execution techniques. 

For example,
- Security log shows a login
- Sysmon shows a suspicious script execution
- Task Scheduler log shows a new scheduled task created

That combination strongly suggests persistence has been established.

## Key Event IDs
- 106 (Task registered (created))
- 140 (Task updated)
- 141 (Task deleted)
- 200 (Task started)
- 201 (Task completed)

## Windows Defender Operational Log

```bash
[WinEventLog://Microsoft-Windows-Windows Defender/Operational]
disabled = 0
index = main
```

The Windows Defender Operational log records activity from the built-in antivirus and endpoint protection system in Windows. This log captures detections, scans, remediation actions, and status changes related to malware protection on the endpoint.

Examples of things that may appear here:
- Malware detections and alerts
- Quarantine or remediation actions
- Real-time protection events
- Scan results (quick/full scans)
- Definition updates
- Protection being enabled or disabled
- Tampering attempts against Defender

## Why this log matters

This log is important because it provides direct evidence of malicious activity detected by the system's antivirus. Unlike other logs that require correlation and interpretation, this log can sometimes give you a clear signal that something malicious was identified.

This log allows you to detect:
- Malware execution attempts
- Files being flagged as malicious
- Real-time protection triggering
- Defender being disabled or tampered with
- Failed or skipped scans

## SOC value

From a SOC perspective, This is a high-signal log source, but it should not be blindly trusted as the only source of truth.

For example,
- Defender detects malware (strong indicator of compromise)
- But no Defender alert does NOT mean the system is clean

This log is best used in combination with others. For example,
- Sysmon shows suspicious file execution
- PowerShell log shows download command
- Defender log shows malware detection

That combination confirms a real threat.

## Common Event IDs

- 1116 (Malware detected)
- 1117 (Malware action taken (cleaned/quarantined))
- 1118 (Malware action failed)
- 5001 (Real-time protection disabled)
- 5007 (Defender configuration changed)

## Setup Log

```bash
[WinEventLog://Setup]
disabled = 0
index = main
```

This Windows Setup log records events related to system installation, updates, upgrades, and major configuration changes. This includes Windows updates, feature upgrades, driver installations, and changes made during system setup or servicing. This log focuses on system changes at a configuration and update level, rather than user activity or application behavior.

Examples of things that may appear here:
- Windows Update installations
- Feature updates and upgrades
- Driver installations and changes
- System setup or configuration events
- Installation of system components
- Changes made during OS servicing

## Why this log matters

The Setup log is important because attackers may attempt to blend in with legitimate system changes or abuse installation mechanisms. While this log is not typically a primary detection source, it can help identify:
- Unexpected system changes
- Suspicious installations
- Driver changes that could indicate malicious behavior
- Timeline context during an investigation

## SOC value

From a SOC perspective, the Setup log is mainly used as context and supporting evidence rather than a primary alert source

For example,
- System shows suspicious behavior
- Setup log shows a driver or update was installed shortly before
- That may help explain or correlate activity

In some cases, attackers may install malicious drivers or abuse update mechanisms, and this log can help reveal those changes.

## WMI Activity Operational Log

```bash
[WinEventLog://Microsoft-Windows-WMI-Activity/Operational]
disabled = 0
index = main
```

The WMI (Windows Management Instrumentation) Activity Operational log records activity related to WMI queries, method executions, and remote interactions with the system. WMI is a built-in Windows framework used for system management and automation. It allows administrators (and attackers) to query system information, execute commands, and interact with remote machines.

Examples of things that may appear here:
- WMI queries executed locally or remotely
- Remote command execution via WMI
- Process creation triggered through WMI
- Access to system information via WMI queries
- Errors or failures in WMI operations

## Why this log matters

WMI is commonly abused by attackers because:
- It is built into Windows
- It can execute commands remotely
- It often bypasses traditional detection methods
- It blends in with legitimate administrative activity

Attackers use WMI for:
- Lateral movement
- Remote command execution
- Reconnaissance (querying system info)
- Persistence (via WMI event subscriptions)

This log provides visibility into those actions.

## SOC value

From a SOC perspective, this is a high-value log for detecting stealthy activity, especially lateral movement.

For example,
- Security log shows login to another machine
- WMI log shows remote query execution
- Sysmon shows process execution on target

That combination strongly indicates lateral movement.

## Key Event IDs
- 5857 (WMI provider started)
- 5858 (WMI operation failed)
- 5859 (WMI operation completed)
- 5860/5861 (WMI method execution)

## Terminal Services - Local Session Manager Log

```bash
[WinEventLog://Microsoft-Windows-TerminalServices-LocalSessionManager/Operational]
disabled = 0
index = main
```

The Terminal Services Local Session Manager log records events related to local and remote session creation, connection, and disconnection on the system. This includes user logins via Remote Desktop Protocol (RDP) and session lifecycle activity. This log focus on session-level activity, meaning how ussers connect to and interact with the system through sessions.

Examples of things that may appear here:
- RDP session creation
- User session logon/logoff
- Session reconnection
- Session disconnection
- Session ID tracking
- Local vs remote session distinctions

## Why this log matters

Remote Desktop Protocol (RDP) is one of the most commonly used methods for:
- Legitimate remote administration
- Attacker lateral movement
- Persistence through remote access

This log provides visibility into who is connecting to the system via session, especially remote ones.

## SOC value

From a SOC perspective, this is a high-value log for tracking RDP activity and lateral movement. 

For example,
- Security log shows successful login (4624, Logon Type 10)
- Terminal Services log confirm RDP session creation
- Sysmon shows commands executed after login

That combination clearly indicates remote access followed by activity.

## Key Event IDs
- 21 (Session logon (successful session creation))
- 24 (Session disconnected)
- 25 (Session reconnected)

## Terminal Services - Remote Connection Manager Log

```bash
[WinEventLog://Microsoft-Windows-TerminalServices-RemoteConnectionManager/Operational]
disabled = 0
index = main
```

The Terminal Services Remote Connection Manager log records events related to incoming Remote Desktop (RDP) connection attempts to the system. It focuses specifically on the network-level connection phase of RDP before or during authentication. This log is different from the Local Session Manager log.
- RemoteConnectionManager (Connection attempts (network-level))
- LocalSessionManager (Session creation and lifecycle)

Examples of things that may appear here:
- Incoming RDP connection attempts
- Successful RDP connections
- Failed RDP authentication attempts
- Network-level connection errors
- Source system attempting to connect

## Why this log matters

This log is important because it shows who is trying to connect to your system over RDP, even before a full session is created. Attackers commonly use RDP for:
- Brute force attacks
- Credential stuffing
- Lateral movement
- Remote persistence

This log allows you to detect:
- Repeated RDP connection attempts
- Failed login attempts over RDP
- Suspicious external or internal sources connecting
- Early signs of brute force attacks

## SOC value

From a SOC perspective, this log is extremely useful for identifying initial access attempts over RDP.

For example,
- RemoteConnectionManager log shows repeated connection attempts
- Security log shows failed logins (4625)
- LocalSessionManager shows eventual session creation

That sequence indicates brute force, successful compromise, and remote access.

## Key Event IDs
- 1149 (Remote Desktop Services: User authentication succeeded)
